--------------------------------------------------------------------------------
1. PROJECT OVERVIEW
--------------------------------------------------------------------------------

Name        : MOV Stay
Type        : Full-Stack Web Application (MERN Stack)
Purpose     : A PG (Paying Guest) / Hostel Management Platform designed for
              property owners (landlords) to manage their PG listings, rooms,
              availability, and view analytics — all from a single dashboard.

Target Users:
  - Owners     : Manage PG properties, rooms, availability status.
  - Students   : Book/view PG accommodation (data exists in DB, future feature).
  - Admins     : Superusers with full access (future feature).

Hosted DB    : MongoDB Atlas (cluster: ai-rehab-admin.jdodnnl.mongodb.net)
               Database Name: mov_stay

Tech Stack:
  Frontend  : React (Vite), Material UI (MUI), Recharts, React Router v6
  Backend   : Node.js, Express.js v5
  Database  : MongoDB (Mongoose ODM)
  Auth      : JWT (JSON Web Tokens) + bcrypt password hashing
  Storage   : Cloudinary (image uploads via Multer)
  AI Data   : @faker-js/faker (synthetic data seeder)

--------------------------------------------------------------------------------
3. BACKEND API STRUCTURE
--------------------------------------------------------------------------------

Server runs on: http://localhost:5000

API Routes:
  /api/auth         → authRoutes.js        (register, login, JWT token)
  /api/pg           → pgRoutes.js          (CRUD for PG listings)
  /api/rooms        → roomRoutes.js        (CRUD for rooms, search)
  /api/availability → availabilityRoutes.js (read/update bed availability)
  /api/dashboard    → dashboardRoutes.js   (owner stats, recent activity)
  /api/upload       → uploadRoutes.js      (Cloudinary image uploads)

Authentication:
  - JWT token stored on client (localStorage or context)
  - authMiddleware.js validates Bearer token on protected routes
  - Roles: student | owner | admin

4. DATABASE MODELS (15 Mongoose Schemas)
--------------------------------------------------------------------------------

Core Models:
  User              → name, email, password (hashed), role, phone, profileImage
  PgListing         → pgName, address, location, neighborhoodId, ownerId,
                      rent, genderPreference, amenities{}, images[], description,
                      isActive
  PgRoom            → pgId (ref), roomType (Single/Double/Triple/Dormitory),
                      sharingCapacity, totalBeds, roomRent, acAvailable
  PgAvailability    → roomId (ref), availableBeds, status (Available/Limited/Full),
                      lastUpdatedOn
  PgReview          → pgId, studentId, rating (1-5), reviewText, sentimentScore

Supporting Models:
  Neighborhood      → locationName, city, safetyScore, transportScore,
                      convenienceScore, lifestyleScore, environmentScore,
                      averageRent, nearbyHospitals[], nearbyColleges[],
                      nearbyTransport[], popularAmenities[]
  StudentProfile    → userId (ref), preferences, budget, preferredLocations[]
  RoommatePreference → userId, preferences object
  RoommateMatch     → user1Id, user2Id, compatibilityScore
  VisitBooking      → studentId, pgId, scheduledDate, status
  Favorite          → studentId, pgId
  Complaint         → userId, pgId, complaintText, status
  Notification      → userId, message, isRead, createdAt
  RecommendationLog → studentId, recommendedPgIds[], algorithm used
  Owner             → userId (ref), additional owner profile fields

--------------------------------------------------------------------------------
7. AUTHENTICATION FLOW
--------------------------------------------------------------------------------

1. User visits any protected route → ProtectedRoute checks AuthContext
2. If not logged in → redirect to /login
3. User enters credentials → POST /api/auth/login
4. Server validates, returns JWT token + user object
5. AuthContext stores user + token (persisted likely in localStorage)
6. All API calls from api.js attach Authorization: Bearer <token> header
7. Backend authMiddleware.js verify token → attach user to req.user
8. Owner-specific routes filter data by req.user._id (ownerId)

Roles defined:
  student  : Can browse PGs, book visits, write reviews (future pages)
  owner    : Full CRUD on their PGs, rooms, availability
  admin    : Full database access (future implementation)

8. SYNTHETIC AI DATA (seedAIData.js)
--------------------------------------------------------------------------------

Purpose: Generate realistic synthetic data for AI/ML model training.

Generated Collections:
  Users         : 500 total
                   - 400 students  (role: "student")
                   -  80 owners    (role: "owner")
                   -  20 admins    (role: "admin")
                  All passwords: "password123" (bcrypt hashed)

  Neighborhoods : 10 areas in Chennai:
                  Velachery, Adyar, Taramani, Guindy, Tambaram,
                  Anna Nagar, Perungudi, Sholinganallur, Chromepet, Medavakkam
                  Each with: safety/transport/convenience/lifestyle/env scores,
                  averageRent, hospitals, colleges, transport, amenities

  PgListings    : 150 PG properties
                  Random owner assigned, random neighborhood, rent ₹4k-₹14k
                  Amenities: WiFi, food, laundry, parking, powerBackup, CCTV,
                  gym, studyRoom (mix of true/false)

  PgRooms       : 4 rooms per PG × 150 PGs = 600 rooms
                  Types: Single(₹9k-₹14k), Double(₹7k-₹10k),
                         Triple(₹5k-₹8k), Dormitory(₹3.5k-₹5.5k)

  PgAvailability: 1 record per room = 600 records
                  Occupancy by rent: <₹5k→90%, <₹9k→70%, else→50%
                  Status: Available | Limited | Full

  PgReviews     : 1200 reviews
                  Rating 1-5, sentiment score = rating/5
                  Text templates: positive (≥4), average (3), negative (<3)

10. AGENTIC AI INTEGRATION (Planned / In-Progress)
--------------------------------------------------------------------------------

The project is being developed with Agentic AI features for:

Core AI Use Cases (based on models present):
  1. Personalized PG Recommendations
     - Model: RecommendationLog
     - Input: StudentProfile (budget, location preferences)
     - Output: Top PG matches with scoring

  2. Neighborhood Intelligence
     - Model: Neighborhood (multi-dimensional scores)
     - Used for: location-based filtering + AI ranking

  3. Roommate Matching
     - Models: RoommatePreference, RoommateMatch
     - AI matches students by preference compatibility scores

  4. Sentiment Analysis on Reviews
     - Model: PgReview (sentimentScore field)
     - Used to rank PG quality from student reviews

  5. Occupancy Prediction
     - Models: PgAvailability, PgRoom (rent tiers)
     - Synthetic occupancy rates already encoded in seeded data

AI-Ready Dataset Fields:
  - sentimentScore (review sentiment 0.0–1.0)
  - safetyScore, transportScore, convenienceScore (neighborhood quality)
  - occupancyRate (availability status → occupancy pattern)
